# 皇冠赛量化指标聚合梳理

> 目标：基于当前已经落库的飞书数据表，明确哪些评分指标现在可以自动聚合，怎么聚合，哪些还缺口径或字段。

---

## 1. 结论先看

当前可以把指标分成三类：

### 1.1 可以直接自动聚合

这些指标用现有表结构就能先跑出第一版：

- 研发岗：`消耗需求评估标准工时`
- 产品岗：`产品需求使用研发测试PD数`
- 产品/设计/研发：`线上问题系统解决数`
- 产品/设计：`需求变更消耗PD`、`设计返工消耗PD`
- 研发：`过程问题(P0/P1/P2)`、`线上故障(P1/P2)`、`线上问题(仅研发代码)` 的第一版

### 1.2 可以半自动聚合，但口径要先定清楚

- 产品：`需求评审通过准时率`
- 产品：`核心项目准时上线率`
- 设计：`设计需求完成数（加权）`
- 设计：`设计交付准时率`
- 研发：`提测不通过`

### 1.3 当前不适合自动聚合，建议继续人工或举证

- 所有 `admin` 来源指标
- 所有 `evidence` 来源指标
- 依赖线下评审、打分表、外部表单、微社区接口的数据项

---

## 2. 聚合的统一口径

所有自动聚合指标，建议统一遵守下面四个步骤。

### 2.1 先限定赛季时间范围

以 `seasons.start_date ~ seasons.end_date` 为准。

不同表优先使用的时间字段建议如下：

- 工时表 `feishu_workitem_gongshi`：优先 `work_date`，其次 `create_time`
- 需求表 `feishu_workitem_story`：完成类用 `finish_time`，新增类用 `start_time`
- 项目表 `feishu_workitem_project`：优先 `start_time` 或后续补充的上线时间字段
- 缺陷表 `feishu_workitem_issue`：优先 `archiving_date`，未归档时退化到 `start_time`

### 2.2 先做人员归属

原始飞书表里多是 `user_key`，本地赛季成员是 `users.id`。

推荐映射链路：

1. 原始表中的 `owner / resolver / work_hour_reporter`
2. 关联 `feishu_user.user_key`
3. 通过 `feishu_user.email = users.email` 映射到本地用户
4. 再关联到 `season_members`

这一步建议沉淀成统一用户映射视图或统一查询片段，不要每个指标单独拼。

### 2.3 再按指标规则聚合

聚合后的结果统一写入 `season_indicator_scores.raw_value`。

### 2.4 最后做分数换算

- `threshold` 指标：根据 `threshold_100 / threshold_60` 算 `threshold_score`
- `deduction` 指标：根据 `deduction_per_unit / deduction_divisor / deduction_cap` 算 `final_score`

## 3. 各岗位量化指标聚合建议

## 3.1 产品岗

### 指标：产品需求使用研发测试PD数

来源：

- `feishu_workitem_gongshi`

建议归属人：

- 归属给需求负责人，优先 `feishu_workitem_story.current_status_operator`
- 如果口径更偏“需求创建责任人”，也可用 `story.owner`
- 这点要先在业务上定死

建议时间范围：

- 取工时表 `work_date`

建议聚合逻辑：

1. 工时表按 `related_requirement` 关联到需求表 `feishu_workitem_story.work_item_id`
2. 找到每个需求对应的产品负责人
3. 汇总该需求在赛季内消耗的 `pd_count`
4. 按产品负责人汇总求和

建议 raw_value：

- 直接存赛季内累计 `pd_count`

注意点：

- 规则文档里提到“客户端 3 PD 折算 1 PD”，当前表里没有明确“客户端/非客户端”的标准化字段，若要严格实现需补口径
- 如果 `related_requirement` 里不是单值 ID，而是字符串或多值，落地时要先确认格式

第一版可行性：

- 可做第一版

---

### 指标：需求评审通过准时率

来源：

- 当前库内无完整评分所需字段

现有表缺口：

- 没有“计划评审通过时间”
- 没有“实际评审通过时间”
- 没有“评审是否通过”的结构化字段

建议：

- 暂时人工录入
- 或补一张评审结果同步表后再自动化

第一版可行性：

- 不能直接自动聚合

---

### 指标：核心项目准时上线率

来源：

- 当前项目表 `feishu_workitem_project`

现有表缺口：

- 没有明确“计划上线时间”
- 没有明确“实际上线时间”
- 没有明确“核心项目标识”或“上线完成标识”

如果要半自动，至少需要补这些字段中的两个：

- `planned_launch_time`
- `actual_launch_time`
- `is_core_project`

第一版可行性：

- 当前不建议自动化

---

### 指标：需求变更消耗PD

来源：

- `feishu_workitem_gongshi`
- `feishu_workitem_story`

建议归属人：

- 跟“产品需求使用研发测试PD数”一致，归属给需求负责人

建议聚合逻辑：

1. 从工时表筛选与需求变更相关的工时
2. 赛季内按需求负责人汇总 `pd_count`
3. 聚合结果写入 `raw_value`
4. 最终按规则 `每 3 PD 扣 1 分，封顶 15`

当前最大问题：

- “需求变更”目前没有明确标准化字段
- “业务发起变更不计入扣分”也没有字段支持

建议第一版：

- 先引入人工标记规则，例如用 `specific_work_hour_type` 或 `description` 关键词筛选
- 但这只是过渡方案，准确性有限

第一版可行性：

- 能做，但口径风险高

---

### 指标：线上问题系统解决数

来源：

- `feishu_workitem_issue`

建议归属人：

- 优先 `resolver`

建议时间范围：

- 优先 `archiving_date`

建议聚合逻辑：

1. 筛选赛季内已归档的问题
2. 归属到 `resolver`
3. 只统计“系统解决”的问题
4. 按人计数

当前最大问题：

- “系统解决”的定义还没结构化
- “自己在两个季度内上线后产生的问题不计算”当前库里也无法识别

建议第一版：

- 先以 `resolver` 在赛季内解决的问题数作为代理值
- 后续补过滤条件

第一版可行性：

- 可做第一版代理口径

---

## 3.2 设计岗

### 指标：设计需求完成数（加权）

来源：

- `feishu_workitem_story`

建议归属人：

- 优先 `current_status_operator`

建议时间范围：

- 优先 `finish_time`

建议聚合逻辑：

1. 取赛季内完成的需求
2. 归属给设计负责人
3. 按复杂度加权后求和

当前最大问题：

- 当前表里没有“复杂度等级”字段
- `template_type` 可能能部分代理需求类型，但不能直接等价为复杂度

建议第一版：

- 如果业务能提供“类型 -> 复杂度权重”映射，可先按 `template_type` 做代理映射
- 否则先不要自动化

第一版可行性：

- 半自动，需要补一张映射表或补字段

---

### 指标：设计交付准时率

来源：

- `feishu_workitem_story`

现有表缺口：

- 没有计划交付时间
- 没有最终通过时间

第一版可行性：

- 当前不建议自动化

---

### 指标：设计返工消耗PD

来源：

- `feishu_workitem_gongshi`
- `feishu_workitem_story`

建议聚合逻辑：

1. 取赛季内与设计返工相关的工时
2. 通过 `related_requirement` 关联到需求
3. 归属给对应设计负责人
4. 汇总 `pd_count`
5. 按 `每 3 PD 扣 1 分，封顶 15`

当前最大问题：

- 没有明确“设计返工”的结构化字段

建议第一版：

- 用 `specific_work_hour_type`、`description`、`work_content_description` 做关键词命中
- 但这仍是过渡口径

第一版可行性：

- 能做第一版，但风险高

与产品岗共用同一套逻辑。

第一版可行性：

- 可做代理口径

## 3.3 研发岗

### 指标：消耗需求评估标准工时

来源：

- `feishu_workitem_gongshi`

建议归属人：

- `work_hour_reporter`

建议时间范围：

- `work_date`

建议聚合逻辑：

1. 筛选赛季内工时记录
2. 按研发人员汇总 `actual_work_hours`
3. 得到 `raw_value`

第一版可行性：

- 直接可做

这是目前最稳、最容易落地的自动化指标。

---

### 指标：过程问题(P0/P1/P2)

来源：

- `feishu_workitem_issue`

建议归属人：

- 如果这是研发质量指标，建议归属给 `resolver`

建议时间范围：

- 优先 `archiving_date`

建议聚合逻辑：

1. 取赛季内归档的问题
2. 归属给研发负责人
3. 按 `severity` 分级计数：
   - P0 数量
   - P1 数量
   - P2 数量
4. 再根据产品规则换算到 `threshold_score`

当前问题：

- 你当前 `scoring_dimensions` 只支持单个 `raw_value`
- 但这个规则实际依赖 P0/P1/P2 三个值联合判断

建议：

- 第一版把 `raw_value` 存成综合风险值并不理想
- 更合理是把判分逻辑直接写进该指标聚合器，在 `calc_snapshot_json` 里保存三档计数

推荐快照内容：

```json
{
  "p0_count": 0,
  "p1_count": 1,
  "p2_count": 6
}
```

第一版可行性：

- 可以自动算，但这类指标不适合只看单个 `raw_value`

---

### 指标：线上故障(P1/P2)

来源：

- `feishu_workitem_issue`

建议归属人：

- 优先 `resolver`

建议聚合逻辑：

1. 取赛季内归档的问题
2. 只取 `is_online_defect` 表示线上故障的数据
3. 严重级别筛出 P1/P2
4. 按人计数
5. 按规则计算扣分

当前问题：

- 文档规则有 P1/P2/P3/P4 版本，你当前种子数据只配置成 `线上故障(P1/P2)`
- 说明业务规则和代码种子之间还没完全对齐

第一版可行性：

- 可以自动算，但要先确认严重级别口径

---

### 指标：线上问题(仅研发代码)

来源：

- `feishu_workitem_issue`

建议归属人：

- `resolver`

建议聚合逻辑：

1. 取赛季内归档的问题
2. 根据 `root_cause` 或 `source` 判断是否属于研发代码问题
3. 按严重级别计数
4. 按规则扣分

当前问题：

- “仅研发代码”需要稳定枚举值映射

建议第一版：

- 先维护一个代码问题枚举白名单
- 聚合器中按白名单过滤

第一版可行性：

- 可以自动算第一版

## 4. 建议优先实现的自动化指标

如果目标是先把自动算分链路跑起来，建议只做第一批最稳的指标。

### 第一批：低风险、可直接上线

- 研发：`消耗需求评估标准工时`
- 产品：`产品需求使用研发测试PD数`
- 各岗位：`线上问题系统解决数` 第一版代理值
- 研发：`线上故障(P1/P2)` 第一版
- 研发：`线上问题(仅研发代码)` 第一版

### 第二批：能做但要补规则映射

- 设计：`设计需求完成数（加权）`
- 研发：`过程问题(P0/P1/P2)`
- 产品/设计：`需求变更消耗PD`、`设计返工消耗PD`

### 第三批：继续人工或后补数据源

- 需求评审通过准时率
- 核心项目准时上线率
- 设计交付准时率
- 提测不通过

---

## 5. 推荐的聚合实现方式

不要把所有指标塞进一个大 SQL。

建议采用“指标聚合器注册表”：

- 一个指标一个聚合函数
- 输入：`season`, `seasonMember`, `scoringDimension`
- 输出：
  - `raw_value`
  - `threshold_score`
  - `final_score`
  - `calc_snapshot_json`

这样做的好处：

- 某个指标规则改了，只改一个函数
- 特殊指标可以写自定义逻辑
- 方便逐步落地，而不是一次性吃掉所有规则

---

## 6. 最终建议

基于当前落库情况，最合理的推进方式是：

1. 先把结果表建好
2. 先落第一批能稳定自动算的指标
3. 其余指标明确标成“人工”或“待补字段”
4. 不要为了全自动而用大量模糊关键词硬凑规则

当前最适合优先实现的核心链路是：

- 工时表 -> 研发工时指标
- 工时表 + 需求表 -> 产品 PD 指标
- 缺陷表 -> 研发问题类指标
- 缺陷表 -> 各岗位问题解决数代理指标

这样可以先把“维度分 + 总分”的主链路跑通，再逐步补齐细口径。
